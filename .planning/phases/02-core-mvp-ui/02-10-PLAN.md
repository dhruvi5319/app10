**Root cause (from VERIFICATION.md):**
`uploadFile()` resolves after `startSSETracking(docId)` is called (line 222) â€” before SSE delivers any stage updates. SSE stage updates go to `documents[]` state (DocumentCard's `StatusBadge`), NOT back to `UploadZone.inFlightFiles`. So `FileProgressBar` only ever sees `UPLOADING` then `READY`.

**Fix strategy: Add `onStageUpdate` callback to `uploadFile`**

The cleanest fix is to pass an optional callback from `UploadZone` into `uploadFile()` that fires on each SSE event, letting `UploadZone` update its local `inFlightFiles` state for the in-progress file.

### Changes to `frontend/src/hooks/useDocuments.ts`


   ```typescript

   ```

<task type="auto">
  <name>Task 2: Add fade-out animation before DocumentCard is removed from DOM</name>
  <files>frontend/src/components/documents/DocumentCard.tsx</files>
  <action>
**Root cause (from VERIFICATION.md):**
`DocumentCard` has `transition: 'opacity 0.3s ease'` on the card div, but the transition is only useful for hover/active state changes. Actual deletion calls `setDocuments(prev => prev.filter(...))` synchronously â€” the card is removed from DOM immediately with no exit animation.

**Fix: `isDeleting` state flag + setTimeout before calling `onDelete`**

This approach requires no new library (no framer-motion, no react-transition-group). It is the simplest correct solution.

### Changes to `frontend/src/components/documents/DocumentCard.tsx`

1. Add `isDeleting` state:
   ```typescript
   const [isDeleting, setIsDeleting] = useState(false);
   ```

2. Change `handleDeleteConfirm` to trigger the fade before deleting:
   ```typescript
   const handleDeleteConfirm = async () => {
     setShowDeleteDialog(false);
     setIsDeleting(true);                         // start opacity fade
     await new Promise((resolve) => setTimeout(resolve, 300)); // wait for CSS transition
     await onDelete(doc.doc_id);                  // now remove from parent state
   };
   ```

3. Apply `isDeleting` to the card's inline style so the transition fires:
   ```typescript
   <div
     className="card"
     style={{
       display: 'flex',
       alignItems: 'flex-start',
       gap: 10,
       padding: '10px 12px',
       marginBottom: 6,
       transition: 'opacity 0.3s ease',
       opacity: isDeleting ? 0 : 1,           // NEW: drives the exit animation
     }}
   >
   ```

The existing `transition: 'opacity 0.3s ease'` on the card div is already in place â€” the only missing piece was setting `opacity` to `0` before removal. With `isDeleting` toggled to `true`, React triggers a re-render that sets `opacity: 0`, the CSS transition runs for 300ms, and then `onDelete` fires to remove the card from parent state.

**Edge case: double-click / unmount during animation**
The `await new Promise(setTimeout)` is fire-and-forget from the perspective of the component lifecycle. If the component unmounts during the 300ms wait (e.g., parent refreshes documents), the subsequent `await onDelete(...)` may throw or be a no-op â€” this is acceptable (the card is already gone). No cleanup ref is needed for this short timeout.
